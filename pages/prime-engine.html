Prime Engine — Engine Systems Implementation

Role: Engine Programmer
Tool: C++, Prime Engine

What

This project involved modifying and extending Prime Engine, a minimal C++ game engine used for gameplay and rendering experiments.

I implemented several engine-level systems, including:

  • View frustum culling for rendering optimization
  • Hierarchical BVH culling (static median-split and dynamic Morton-code trees)
  • Collision detection and physics movement (move-and-slide with gravity)
  • Skeletal animation blending (full-body, partial-body, and additive layers)
  • Debug visualization tools for bounding volumes and animation state

Working inside the engine required understanding its component system, event system, and scene graph structure in order to integrate new features without breaking the existing architecture.

How

Frustum Culling

To reduce unnecessary rendering work, I implemented view frustum culling using bounding boxes.

Steps:

  • Compute the world-space bounding box of each mesh instance.
  • Transform the box corners into clip space using the projection–view matrix.
  • Test the box against the six planes of the camera frustum.
  • If all corners lie outside any plane, the object is marked as culled and skipped during rendering.

I also added debug rendering for bounding boxes to visually verify the correctness of the culling logic. This significantly reduced the number of objects sent to the renderer when many objects were outside the camera view.

Physics and Collision System

I implemented a simple collider system supporting sphere and box colliders attached to game objects.

Each frame:

  • Colliders update their world-space position from the owning object's transform.
  • When an object attempts to move, the engine checks for collisions at the candidate position.
  • If collisions occur, the movement vector is projected onto the collision plane, removing the component that pushes into the surface.
  • The remaining movement causes the object to slide naturally along obstacles.

Gravity

For objects affected by gravity:

  • Vertical velocity is updated each frame using gravitational acceleration.
  • The vertical component of movement is applied to the object.
  • If ground support is detected, vertical velocity resets to prevent continuous downward acceleration.

This produces natural falling behavior while preventing objects from penetrating the ground.

Animation Blending

I extended the soldier's animation state machine with several blending modes, chosen at spawn via an npcType value from Lua. The skeleton is split into an upper body (joints 2–37) and a lower body (joints 38–47), so clips can drive different parts at once.

  • Full-body blend — two clips mixed by a factor (e.g. idle + run, weighted 0.5 each).
  • Partial blend — a clip overrides only a joint range via PARTIAL_BODY_ANIMATION (e.g. upper-body shoot over a full-body idle).
  • Additive blend — a clip layered on top of the existing pose instead of replacing it.

For additive motion I added an ADDITIVE slot flag. The additive pass takes the clip's delta from its reference pose (frame 0) as a rotation-only quaternion and pre-multiplies it onto the joint's local transform — so a base motion stays intact while extra movement is layered on (e.g. lower-body run + upper-body shoot, running and aiming at once).

Hierarchical Culling — Static & Dynamic BVH

I extended frustum culling into a Bounding Volume Hierarchy managed by a central CullingManager, so large groups of objects can be rejected with a few AABB-vs-frustum tests instead of one test per mesh. Each instance is bucketed as static or dynamic.

  • Static BVH — built once, recursively median-split along the widest centroid axis, with leaves holding ≤ 8 instances. Traversal prunes whole subtrees whose bounds miss the frustum.
  • Dynamic BVH — rebuilt every frame for moving objects (to stress-test it I spawned many small meshes given simple random movement). Centroids are normalized and encoded as 30-bit Morton codes (Z-order), sorted, then split at the midpoint — a linear, cache-friendly construction.

The static tree gives better quality for buildings and terrain, while the Morton-code rebuild is cheap enough to run per frame for many moving units — Z-order keeps spatially close objects adjacent in memory, so the build and traversal access nodes nearly linearly, staying cache-friendly. Both are roughly O(n log n), dominated by the sort.

I implemented several engine-level systems, including:我实现了多个引擎级系统，包括：

  • 用于渲染优化的视锥剔除
  • Hierarchical BVH culling (static median-split and dynamic Morton-code trees)
  • 层级式 BVH 剔除（静态中位数划分树与动态 Morton 编码树）
  • Collision detection and physics movement (move-and-slide with gravity)
  • 碰撞检测与物理运动（带重力的移动-滑动）
  • Skeletal animation blending (full-body, partial-body, and additive layers)

  • Debug visualization tools for bounding volumes and animation state
  • 包围体与动画状态的调试可视化工具

Working inside the engine required understanding its component system, event system, and scene graph structure in order to integrate new features without breaking the existing architecture.要在引擎内部动手，得先摸清它的组件系统、事件系统和场景图结构，才能在不破坏原有架构的前提下加入新功能。

How如何实现

Frustum Culling视锥剔除

To reduce unnecessary rendering work, I implemented view frustum culling using bounding boxes.为减少不必要的渲染开销，我使用包围盒实现了视锥剔除。

Steps:步骤：

  • Compute the world-space bounding box of each mesh instance.
  • 计算每个网格实例的世界空间包围盒。
  • Transform the box corners into clip space using the projection-view matrix.
  • 使用投影-视图矩阵将包围盒顶点变换到裁剪空间。
  • Test the box against the six planes of the camera frustum.
  • 将包围盒与相机视锥的六个平面进行测试。
  • If all corners lie outside any plane, the object is marked as culled and skipped during rendering.
  • 若所有顶点都位于某一平面之外，则该物体被标记为剔除并在渲染时跳过。

I also added debug rendering for bounding boxes to visually verify the correctness of the culling logic. This significantly reduced the number of objects sent to the renderer when many objects were outside the camera view.我还为包围盒添加了调试渲染，以直观验证剔除逻辑的正确性。当大量物体位于相机视野之外时，这显著减少了送入渲染器的物体数量。

Physics and Collision System物理与碰撞系统

I implemented a simple collider system supporting sphere and box colliders attached to game objects.我实现了一个简单的碰撞体系统，支持附加在游戏对象上的球体与盒体碰撞器。

Each frame:每一帧：

  • Colliders update their world-space position from the owning object's transform.
  • 碰撞体根据所属对象的变换更新其世界空间位置。
  • When an object attempts to move, the engine checks for collisions at the candidate position.
  • 当对象尝试移动时，引擎会在候选位置检测碰撞。
  • If collisions occur, the movement vector is projected onto the collision plane, removing the component that pushes into the surface.
  • 若发生碰撞，则将移动向量投影到碰撞平面上，去除指向表面的分量。
  • The remaining movement causes the object to slide naturally along obstacles.
  • 剩余的移动分量使对象沿障碍物自然滑动。

Gravity重力

For objects affected by gravity:对于受重力影响的对象：

  • Vertical velocity is updated each frame using gravitational acceleration.
  • 每帧根据重力加速度更新竖直方向速度。
  • The vertical component of movement is applied to the object.
  • 将竖直方向的移动分量施加到对象上。
  • If ground support is detected, vertical velocity resets to prevent continuous downward acceleration.
  • 若检测到地面支撑，则重置竖直速度，以防止持续向下加速。

This produces natural falling behavior while preventing objects from penetrating the ground.这既产生自然的下落效果，又能防止物体穿透地面。

Animation Blending动画混合

I extended the soldier's animation state machine with several blending modes, chosen at spawn via an npcType value from Lua. The skeleton is split into an upper body (joints 2-37) and a lower body (joints 38-47), so clips can drive different parts at once.我为士兵的动画状态机扩展了多种混合模式，在生成时通过来自 Lua 的 npcType 值选择。骨架被划分为上半身（关节 2-37）与下半身（关节 38-47），因此不同片段可同时驱动不同部位。

  • Full-body blend - two clips mixed by a factor (e.g. idle + run, weighted 0.5 each).
  • 全身混合：按权重混合两个片段（例如待机 + 奔跑，各占 0.5）。
  • Partial blend - a clip overrides only a joint range via PARTIAL_BODY_ANIMATION (e.g. upper-body shoot over a full-body idle).
  • 局部混合：某个片段通过 PARTIAL_BODY_ANIMATION 仅覆盖某一关节范围（例如在全身待机之上叠加上半身射击）。
  • Additive blend - a clip layered on top of the existing pose instead of replacing it.
  • 叠加混合：将某个片段叠加在现有姿势之上，而非替换它。

For additive motion I added an ADDITIVE slot flag. The additive pass takes the clip's delta from its reference pose (frame 0) as a rotation-only quaternion and pre-multiplies it onto the joint's local transform - so a base motion stays intact while extra movement is layered on (e.g. lower-body run + upper-body shoot, running and aiming at once).为实现叠加运动，我增加了一个 ADDITIVE 槽位标志。叠加阶段将片段相对其参考姿势（第 0 帧）的增量作为仅含旋转的四元数，并左乘到关节的局部变换上，这样基础动作保持不变，而额外动作被叠加其上（例如下半身奔跑 + 上半身射击，边跑边瞄准）。

Hierarchical Culling - Static & Dynamic BVH层级式剔除：静态与动态 BVH

I extended frustum culling into a Bounding Volume Hierarchy managed by a central CullingManager, so large groups of objects can be rejected with a few AABB-vs-frustum tests instead of one test per mesh. Each instance is bucketed as static or dynamic.我将视锥剔除扩展为由中央 CullingManager 管理的包围体层级（BVH），从而只需少量 AABB 与视锥的测试即可剔除大批物体，而非对每个网格逐一测试。每个实例被归类为静态或动态。

  • Static BVH - built once, recursively median-split along the widest centroid axis, with leaves holding ≤ 8 instances. Traversal prunes whole subtrees whose bounds miss the frustum.
  • 静态 BVH：一次性构建，沿质心分布最广的轴递归进行中位数划分，叶节点最多容纳 8 个实例。遍历时会剪除包围盒未命中视锥的整棵子树。
  • Dynamic BVH - rebuilt every frame for moving objects (to stress-test it I spawned many small meshes given simple random movement). Centroids are normalized and encoded as 30-bit Morton codes (Z-order), sorted, then split at the midpoint - a linear, cache-friendly construction.
  • 动态 BVH：针对运动物体每帧重建（为了给它加压测试，我生成了大量随机运动的小网格）。质心被归一化并编码为 30 位 Morton 码（Z 序），排序后在中点划分：一种线性、对缓存友好的构建方式。

The static tree gives better quality for buildings and terrain, while the Morton-code rebuild is cheap enough to run per frame for many moving units - Z-order keeps spatially close objects adjacent in memory, so the build and traversal access nodes nearly linearly, staying cache-friendly. Both are roughly O(n log n), dominated by the sort.静态树对建筑与地形能提供更高的质量，而 Morton 码重建足够廉价，可对大量运动单位每帧执行，Z 序使空间上相近的物体在内存中也相邻，因此构建与遍历几乎线性地访问节点，保持缓存友好。两者的复杂度都约为 O(n log n)，主要由排序决定。

Static vs. Dynamic BVH Construction静态与动态 BVH 的构建对比

Static BVH - rainbow boxes from root (red) to leaf (purple).静态 BVH：从根节点（红色）到叶节点（紫色）的彩虹色包围盒。

Dynamic BVH - white boxes over randomly moving meshes, leaves shown as dots.动态 BVH：覆盖在随机运动网格上的白色包围盒，叶节点以圆点表示。
